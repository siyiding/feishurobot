> 版本：v1.0 | 作者：Kiro💻 | 日期：2026-04-02
> 状态：初稿

---

## 1. 排期原则

1. **M1 先跑通机器人基础框架**，不合并到 M2
2. **M3 的推送线和 DR 线并行开发**，按依赖关系串/并
3. **M6 工作量分两批**，避免集中延期
4. **测试工作量不含在开发人天内**（需测试部另行估算 ~20-25人天）
5. **风险预留**：按 1.15 系数估算实际工期 = 105-108 人天

## 3. 详细排期

### M1（第1个月）：模块1 + 模块2

**目标**：机器人跑起来，能查缺陷、能收P0通知

```
第1周
├── 1.1 飞书应用创建（0.5天）→ 申请机器人应用
├── 1.2 后端框架搭建（3天）→ FastAPI + 消息接收
└── 1.3 意图识别基础（2天）→ 三类意图分发

第2周
├── 2.1 飞书项目API接入（3天）→ 缺陷列表/项目列表
├── 2.2 缺陷查询（2天）→ 按项目/状态筛选
└── 1.4 机器人配置（0.5天）→ 名称/头像

第3周
├── 2.3 缺陷创建（2天）→ 说话建缺陷单
└── 2.4 缺陷状态更新（2天）→ 更新状态

第4周
├── 2.5 缺陷变更监听（2天）→ Webhook监听
├── 4.1 Redis队列部署（1天）→ 部署 + 基础队列
└── 4.2 P0即时推送（2天）→ P0不限流急行道

验收：机器人能被@，收到"查XX项目未关闭缺陷"，5秒内返回列表
```

**M1 准入准出**：
- 准入：模块1+2开发完成，自动化测试通过
- 准出：缺陷查询/创建/推送功能正常

---

### M2（第2个月）：模块3（用例库接入）

**目标**：对接现有飞书电子表格用例库，场景库关联

```
第1-2周
├── 3.1 用例库接入（2天）→ 飞书电子表格 API
├── 3.6 用例查询+更新（2天）→ 按类型/模块/状态筛选
└── 3.7 场景覆盖更新（1天）→ 用例执行后自动更新

验收：机器人查询结果与电子表格手动查询100%一致
```

**注意**：已有机用例，无需迁移（省去6人天）

---

### M3（第3个月）：模块4 + 模块6基础

**目标**：推送分级配置，DR数据查询

**并行两条线**：

```
推送线（依赖Redis）          DR线（依赖DR平台API文档）
├── 4.3 P1定时合并推送(2天)  ├── 6.1 DR API接入（5天）
├── 4.4 P2可选推送（1天）    ├── 6.2 信号数据查询（3天）
└── 4.5 推送配置页（3天）    ├── 6.3 静态图表生成（3天）
                              └── 6.4 异常阈值告警（2天）

第4周
├── 4.6 周报摘要推送（1天）→ 每周五17:00
└── 6.5 人工确认流程（2天）→ 确认异常→建缺陷 / 标记噪声
```

**M3 准入准出**：
- 准入：模块4+6基础完成
- 准出：推送接收正常，DR查询返回图表

## 4. 关键路径

```
M1（17天）
  └── M2（5天，模块3依赖M1基础框架）
          └── M3（27天，并行推送线+DR线）
                  └── M4（17天，模块5依赖M3的推送+DR基础）
                          └── M5（7-10天，模块7依赖M3+M4）
                                  └── M6（10-13天，模块8依赖M3+M4）

关键路径总工期（串行）：17+5+27+17+10+13 = 89天 ≈ 5个月
考虑并行压缩：约 5-6 个月
```
