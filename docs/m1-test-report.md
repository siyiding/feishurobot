## 3. 发现的问题（Bug List）

### P1 Bug: BugPriority 未导入导致缺陷更新失败

**严重程度**: P1  
**影响范围**: 缺陷状态更新功能完全不可用  
**复现步骤**:
1. 发送消息："把bug_001状态改成已解决"
2. 返回500错误: `cannot access local variable 'BugPriority' where it is not associated with a value`

**根本原因**:  
在 `app/api/webhook.py` 的 `execute_handler()` 函数中，`update_bug` 分支只导入了 `BugUpdateRequest, BugStatus`，但未导入 `BugPriority`。后续代码引用 `BugPriority.P0` 进行P0推送判断时触发 NameError。

**修复建议**:
```python
# Line ~209: 修复前
elif action == "update_bug":
    from app.models.schemas import BugUpdateRequest, BugStatus
    
# 修复后
elif action == "update_bug":
    from app.models.schemas import BugUpdateRequest, BugStatus, BugPriority
```

**影响评估**: 
- 阻塞缺陷管理核心流程
- 用户无法通过机器人更新缺陷状态
- 必须修复后才能进入M2

---

### P2 Bug: 项目Key提取不准确导致查询结果偏差

**严重程度**: P2  
**影响范围**: 跨项目查询时可能返回错误数据  
**复现步骤**:
1. 发送消息："查ADAS项目未关闭缺陷"
2. 返回的是ICC项目的缺陷（bug_001），而非ADAS项目

**根本原因**:  
在 `app/api/webhook.py` 的 `execute_handler()` 中，当消息包含"未关闭"时，代码强制设置 `req.status = BugStatus.OPEN`，而不再使用 `params["status"]`。这导致查询走的是无project_key过滤的路径，返回所有OPEN状态的缺陷。

**临时规避**:  
用户可以明确指定状态，如："查ADAS项目缺陷状态是open的"

**修复建议**:  
移除强制覆盖逻辑，或在强制设置status时保留project_key过滤。

**影响评估**:
- 查询结果不准确可能误导用户
- 但基本查询功能（ICC项目）正常
- 建议M2修复
