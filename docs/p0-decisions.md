## D1：Bitable 并发写入策略

### 决策问题
多个用户同时操作用例库时，机器人写入是否会冲突？冲突如何处理？

### 推荐方案：机器人只写白名单字段，读可读全字段

```
机器人可写字段（白名单）：
┌────────────────┬────────────────────────────────┐
│ 字段            │ 说明                            │
├────────────────┼────────────────────────────────┤
│ 执行状态        │ 待执行→执行中→通过/失败/阻塞     │
│ 执行日期        │ 最近一次执行时间                  │
│ 执行人          │ 谁执行                          │
│ 关联场景ID      │ 挂住的场景ID列表                  │
│ 最近更新人      │ 谁最后修改                       │
│ 备注            │ 可追加简短说明                   │
└────────────────┴────────────────────────────────┘

机器人只读不回写字段：
  用例ID、用例名称、测试类型、功能模块、关联需求、优先级、用例版本
```

### 理由
- 白名单限制写入范围，避免污染用户数据
- 执行状态/执行人等字段天然适合机器人更新（用户改、机器人也改）
- 其他字段（用例名、类型、优先级）由用户自己维护

### 备选方案
| 方案 | 缺点 |
|------|------|
| 全字段可写 | 风险高，机器人误操作可能破坏数据 |
| 全字段只读 | 无法实现"执行后自动更新状态"，需用户手动改 |

### 冲突处理规则
- Bitable 单字段原子更新，不锁表
- 并发冲突：后者覆盖前者（Bitable API 特性）
- 高频写入场景（如批量更新执行状态）：Redis 消息队列串行化

## D2：对话上下文持久化边界

### 决策问题
30 分钟快照是否足够？什么情况下会丢上下文？

### 推荐方案：30 分钟基准 + 三重保护

| 触发条件 | 行为 | 是否丢数据 |
|---------|------|-----------|
| 10 轮对话 | 立即快照到 Bitable | ❌ 不丢 |
| 30 分钟无交互 | 快照 + 休眠 | ❌ 不丢 |
| 单会话超 2 小时 | 强制快照 + 新会话开始 | ❌ 不丢 |
| 进程重启 | 从 Bitable 恢复（最多 1 次） | ❌ 不丢 |
| 用户明确结束对话 | 快照 + 清理内存 | ❌ 不丢 |
| Bitable API 故障 | 内存降级 + 告警 | ⚠️ M1 期间可接受 |
| 服务器断电 | 最多丢 30 分钟 | ⚠️ 可接受 |

### 理由
- 测试工程师平均操作间隔 > 15 分钟，30 分钟覆盖绝大多数工作间隙
- 三重保护（10 轮/30 分钟/2 小时上限）确保不会因单一触发条件失效而丢数据
- 进程重启恢复是最坏情况兜底，不是日常路径

### 备选方案
| 方案 | 缺点 |
|------|------|
| 5 分钟快照 | 太频繁，API 调用量大，测试人员操作节奏被打断 |
| 不做快照，全存内存 | 进程重启即丢失，不可接受 |
| 永久保存所有上下文 | 存储成本高，且长期上下文膨胀影响模型效果 |

## D3：飞书机器人鉴权模型

### 决策问题
机器人对所有飞书用户开放，还是限白名单？

### 推荐方案：M1-M2 白名单模式，M3 后逐步开放

```
M1-M2（验证期）：
┌─────────────────────────────────┐
│  白名单模式（Redis Hash存储）     │
│  - 只有名单内用户可以@机器人       │
│  - 名单动态管理：@机器人 添加/移除  │
│  - 飞书机器人「可用范围」设为全员   │
│    （机器人侧做二次校验）          │
└─────────────────────────────────┘

M3+（稳定期）：
  - 切换为飞书「可用范围」管控
  - 白名单保留作为「管理员模式」
  - 敏感操作（建缺陷/改状态）保留额外校验
```

### 理由
- M1-M2 是内部验证阶段，用户范围可控，白名单降低风险
- 白名单存在 Redis，管理员可动态调整，无需重启服务
- M3 正式推广时再切换为「可用范围」管控，体验更好

### 备选方案
| 方案 | 缺点 |
|------|------|
| 全开放 | 任何飞书用户都能@机器人，无法管控测试部外人员 |
| 全封闭（管理员审批） | 上线初期反馈慢，不利于快速迭代 |

### 白名单管理命令
```
@机器人 添加用户 张三
@机器人 移除用户 李四
@机器人 查看白名单
```

## D4：M2 验收标准（替代 docx 行数对比）

### 决策问题
原方案「docx 行数对比」已删除，改用哪些指标？

### 推荐方案：字段完整性验证 + 抽查机制

| 验收项 | 验收方法 | 合格标准 |
|-------|---------|---------|
| 字段完整性 | 自动化检查所有必填字段有值 | 必填字段 100% 有值 |
| 用例可查询 | 机器人查询 vs 电子表格手动查询对比 | 抽样 20 条，准确率 > 95% |
| 状态可更新 | 机器人改状态 → 电子表格同步更新 | 抽样 10 条，100% 成功 |
| 迁移差异率 | 有电子表格 vs 声称应迁移数 | 差异 < 5% |
| 用户可用性 | 测试工程师实际使用反馈 | 无阻碍报错 |

> ~~docx 行数对比~~（已删除，不迁移 docx，此项无意义）

### 理由
- 字段完整性验证直接保障数据质量，适合自动化
- 抽样检查兼顾效率与覆盖率
- 用户可用性是最终验收门槛，技术指标通过但体验差仍需返工

### 备选方案
| 方案 | 缺点 |
|------|------|
| 全量数据对比 | 工作量大，1000+ 条用例不现实 |
| 纯用户反馈验收 | 主观性强，可能遗漏边界问题 |

## D5：DR 平台 API 获取时间和 Buffer 计划

### 决策问题
DR 数据什么时候拉取？实时还是定时？Buffer 设多少？

### 推荐方案：混合策略（定时 + 实时 + Buffer）

```
1. 定时批量拉取（离线分析 / 周报用）
   - 每周五拉取本周数据（T-7 到 T-1）
   - 每周一拉取上周数据（确保完整性）
   - 时间窗口：T-7天 到 T-1天

2. 实时查询（用户主动请求）
   - 用户问"ICC 最近 CAN 总线有没有异常"→ 默认查最近 24 小时
   - 可配置时间范围
   - 结果缓存 5 分钟（Redis key: dr:query:{hash}）

3. DR 数据 Buffer（关键！）
   - DR 数据写入有分钟级延迟
   - 查询时间窗口向后扩 10 分钟
   - 示例：用户查"过去 1 小时" → 实际拉 "1 小时 + 10 分钟"

4. 直播盯守场景（选做）
   - 用户说"帮我盯着 DR 数据" → 每 5 分钟轮询
   - 用户说"停止" → 停止轮询
```

### DR API 调用频率估算

| 场景 | 频率 | 备注 |
|-----|------|------|
| 周报生成 | 1 次 / 周 | 手动触发 |
| 日常查询 | ~50 次 / 天 | 工程师主动查询 |
| 图表生成 | ~20 次 / 天 | 每次 1-3 张图 |
| **合计** | **~70 次 / 天** | 低于 API 限额 |

### 理由
- DR 数据不是实时流，有写入延迟，Buffer 10 分钟是保守安全值
- 定时 + 实时混合覆盖离线分析和在线查询两类场景
- 5 分钟查询缓存避免同一数据的重复调用

### 备选方案
| 方案 | 缺点 |
|------|------|
| 实时流（WebSocket） | DR 平台未必支持，推送架构复杂度高 |
| 只拉原始日志 | 违背 DR 平台不提供原始日志的原则（元数据 only） |
| Buffer 0 分钟 | 可能拿到不完整数据（DR 写入有延迟） |
