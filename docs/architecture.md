## 1. 架构概览

### 1.1 系统定位

整车测试助手飞书机器人是一个**飞书集成层**，对接飞书项目（缺陷/任务）、飞书电子表格（用例库/场景库）、DR平台三大数据源，通过飞书机器人提供自然语言查询、报告生成、缺陷管理和DR数据可视化能力。

**核心原则：不自建持久化存储，所有业务数据读写均通过三方API完成。**

### 1.2 模块分层

```
┌─────────────────────────────────────────────────────────────┐
│                    飞书客户端（用户）                        │
└──────────────────────────┬──────────────────────────────────┘
                           │ 飞书开放平台 Webhook / 轮询
┌──────────────────────────▼──────────────────────────────────┐
│                    模块1：飞书机器人基础                       │
│              （消息接收 → 意图识别 → 路由分发）                 │
└────┬─────────────┬──────────────┬──────────────┬────────────┘
     │             │              │              │
┌────▼────┐  ┌─────▼────┐  ┌─────▼─────┐  ┌─────▼─────┐
│ 模块2   │  │ 模块3     │  │ 模块4     │  │ 模块5     │
│ 飞书项目│  │ 电子表格  │  │ 通知推送  │  │ 问答报告  │
│ 数据对接│  │ 用例库    │  │ 系统      │  │          │
└────┬────┘  └─────┬────┘  └─────┬─────┘  └─────┬─────┘

│  │ 消息队列 │  │ 快照存储 │  │  API      │  │  API         │  │
```

### 1.3 两批上线节奏

| 批次 | 里程碑 | 模块 | 上线目标 |
|------|--------|------|----------|

## 2. 模块间调用关系

### 2.1 模块1：飞书机器人基础

**职责**：消息入口，意图识别，路由分发

```
飞书消息 → Webhook接收 → 消息解析 → 意图识别 → 路由分发
                                              │
                          ┌───────────────────┼───────────────────┐
                          │                   │                   │
                       模块2               模块3               模块5
                    (飞书项目)         (用例库)            (问答报告)
```

**意图类型**：
- **Query类**：查缺陷、查用例、查里程 → 读飞书项目/Bitable
- **Action类**：创建缺陷、更新状态 → 写飞书项目
- **Report类**：生成周报/专项报告 → 读飞书项目+Bitable → 写飞书文档

### 2.2 模块2：飞书项目数据对接

**职责**：封装飞书项目API，提供缺陷/任务的读写能力

**调用链路**：
```
模块1/模块5/模块7 → FeishuProjectClient → 飞书开放平台 API
                                       │
                                       ├── 缺陷 CRUD
                                       ├── 任务 CRUD
                                       └── 变更事件监听（WebSocket/轮询）
```

### 2.3 模块3：飞书电子表格用例库

**职责**：封装飞书电子表格API，读写用例库和场景库

**调用链路**：
```
模块5（查询）→ BitableClient → 飞书电子表格 API
                                ├── 按字段筛选记录
                                ├── 更新执行状态
                                └── 关联场景覆盖更新
```

**⚠️ 关键约束**：机器人只写特定字段（见D1决策）

### 2.4 模块4：通知推送系统

**职责**：Redis队列 → 限流 → 飞书消息推送

```
业务事件触发（缺陷变更/DR告警/P0通知）
        │
        ▼
  Redis Stream 队列
  （按用户ID分片）
        │
        ├── P0 → 实时推送（无延迟）
        ├── P1 → 30分钟窗口合并
        └── P2 → 用户自选频率
                │
                ▼
         飞书消息 API
         （每分钟 ≤ 60 条/每人）
```

### 2.5 模块5：自然语言问答+报告生成

**职责**：20个意图模板，数据聚合，报告生成

```
用户自然语言输入
        │
        ▼
  意图匹配（规则模板，非LLM）
        │
        ▼
  多轮上下文加载（内存+Bitable快照）
        │
        ▼
  数据聚合（飞书项目 + Bitable + DR）
        │
        ▼
  响应生成（markdown/结构化文本）
        │
        ▼
  写飞书文档（报告类）→ 返回链接
```

### 2.6 模块6：DR平台对接（CLI模式）

**职责**：CLI 调用 DR 平台，按行程查询问题列表，不解析 Bag 包

```
用户："查一下 ICC 行程最近有没有问题"
        │
        ▼
  subprocess 调用 DR CLI
        │
        ├── dr query --trip <trip_id>
        └── dr tag --info <tag_id>
              │
              ▼
  解析返回的问题列表 + 关联数据
        │
        ▼
  生成自然语言回复 + 跳转链接
        │
        ▼
  推送（模块4）
```

**注意**：matplotlib 图表生成暂不实现（等 DR CLI 返回的数据格式明确后再设计）。

### 2.7 模块7：缺陷管理自动化

**职责**：L1自动建单，L2状态同步，L3（选做）

```
模块5/模块6触发
        │
        ▼
  FeishuProjectClient.create_bug()
        │
        ▼
  状态变更 → 模块4通知相关方
```

### 2.8 模块8：变更感知+回归助手

**职责**：Git MR变更 → 匹配受影响的用例

```
GitLab/GitHub Webhook（MR事件）
        │
        ▼
  变更文件解析 → 模块映射表查询
        │
        ▼
  受影响用例集 → 推荐回归范围
        │
        ▼
  模块4推送 → 测试工程师
```

## 4. 对话上下文存储方案

### 4.1 设计原则

**"内存优先，Bitable为辅"** — 短期上下文放内存，长期/跨会话上下文放Bitable。

### 4.2 上下文存储分层

```
┌─────────────────────────────────────────────────┐
│              Layer 1：进程内存                   │
│  ┌─────────────────────────────────────────┐   │
│  │ 当前会话上下文（Dict）                    │   │
│  │ - key: open_id                          │   │
│  │ - value: [消息历史, 当前意图, 中间状态]    │   │
│  │ - TTL: 会话结束时清理                     │   │
│  └─────────────────────────────────────────┘   │
│                 │ 触发条件                       │
│        ┌────────┼──────────────┐               │
│        │ 10轮对话 │ 30分钟超时   │ 2小时强制     │
│        ▼         ▼              ▼              │
│  ┌──────────────────────────────────────────┐   │
│  │  Layer 2：Bitable 快照                    │   │
│  │  表名：对话上下文快照                      │   │
│  │  - 快照JSON（二进制）                      │   │
│  │  - 对话轮次 / 触发时间 / 所属用户           │   │
│  │  - TTL: 7天后自动归档（删除）              │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 4.3 上下文边界定义

| 触发条件 | 行为 | 说明 |
|---------|------|------|
| 10轮对话 | 立即快照到Bitable | 防止长对话丢失 |
| 30分钟无新消息 | 触发快照，会话休眠 | 典型测试工作间隔 |
| 单会话超过2小时 | 强制快照+新会话开始 | 防止上下文无限膨胀 |
| 进程异常退出 | 启动时从Bitable恢复 | 最多恢复最近1次快照 |
| 用户明确结束对话 | 快照+清理内存 | 例如发送"好了再见" |

### 4.4 Bitable快照表结构

| 字段名 | 类型 | 说明 |
|-------|------|------|
| 快照ID | 文本 | UUID，唯一标识 |
| 用户ID | 文本 | open_id |
| 会话ID | 文本 | 同一会话的快照串联 |
| 触发时间 | 日期时间 | 快照创建时间 |
| 触发原因 | 单选 | 10轮/30分钟/2小时/手动 |
| 上下文JSON | 长文本 | base64编码的对话历史 |
| 对话轮次 | 数字 | 本次快照包含的轮数 |

### 4.5 D2 决策确认：30分钟快照 ✅ 推荐方案

**决策**：30分钟快照足够，附加3个保护规则：
1. ✅ 10轮对话强制快照
2. ✅ 30分钟超时快照（核心逻辑）
3. ✅ 2小时绝对上限（防止内存泄漏）
4. ✅ 进程重启后从Bitable恢复（最多1次）

## 5. 通知推送系统架构

### 5.1 推送分级

| 级别 | 定义 | 推送策略 | 合并规则 |
|------|------|---------|---------|
| P0 | 严重缺陷新增/关键任务超时/DR重大异常 | 即时，零延迟 | 同类型P0合并为1条 |
| P1 | 普通缺陷变更/任务延期 | 每30分钟窗口 | 按类型+用户合并 |
| P2 | 周报摘要/可选通知 | 用户自选 | 不合并 |

### 5.2 Redis队列设计

```
Key: push:queue:{user_id}
Type: Redis Stream

消息格式：
{
  "id": "msg_uuid",
  "level": "P0|P1|P2",
  "type": "bug_new|bug_update|task_overdue|dr_alert|weekly_report",
  "title": "通知标题",
  "content": "通知内容（markdown）",
  "url": "跳转链接（可选）",
  "created_at": "时间戳",
  "enqueue_at": "入队时间"
}
```

### 5.3 限流策略

```
飞书API限制：60次/分/机器人

令牌桶算法：
┌──────────────────────────────────────────┐
│  capacity: 60                            │
│  refill_rate: 60/分钟 → 1条/秒           │
│  当前桶内令牌 → 可用额度                   │
└──────────────────────────────────────────┘

P0特殊处理：
- P0绕过令牌桶，单独急行道（优先队列）
- 但每个P0消息仍需acquire(1)确保不超标
- P0超额时记录告警日志，不丢弃消息
```

### 5.4 推送 Worker 架构

```
┌──────────────────────────────────────────────────────┐
│                   Push Worker（独立进程）              │
│                                                      │
│  while True:                                         │
│    messages = XREADGROUP group worker_N BLOCK 1000   │
│    for msg in messages:                              │
│      if msg.level == "P0":                           │
│        sendImmediately(msg)  # 急行道                │
│      else:                                           │
│        batch.append(msg)  # P1/P2进批次              │
│                                                      │
│    if batch ready (时间到/数量满):                    │
│      deduplicate_and_merge(batch)                    │
│      for user in users:                              │
│        acquire_token()                               │
│        feishu_api.send(user, merged_content)         │
└──────────────────────────────────────────────────────┘
```

### 5.5 同类型P0合并算法

```
场景：同一会话内触发3个P0缺陷通知
输入：[P0_缺陷1, P0_缺陷2, P0_缺陷3]
合并逻辑：
  if 同类型 && 同项目 && 时间窗口 < 5分钟:
    合并为1条："【P0批量告警】XX项目新增3个严重缺陷"
  else:
    各自发送
```

## 6. 模块6 DR接入方案

### 6.1 DR平台定位

DR平台提供**信号元数据**（信号名、时间戳、数值），不提供原始日志文件。

### 6.2 API鉴权方案

```
DR平台认证：API Key / Secret Token

┌─────────────────────────────────────────┐
│  DRAPIClient                            │
│  ┌───────────────────────────────────┐  │
│  │ token_cache: {                    │  │
│  │   access_token: "xxx",           │  │
│  │   expires_at: timestamp,         │  │
│  │   refresh_token: "yyy"           │  │
│  │ }                                 │  │
│  └───────────────────────────────────┘  │
│                                         │
│  def get_token():                       │
│    if token存在 && 未过期:              │
│      return token                       │
│    else:                                │
│      refresh_or_reauth()                │
│      return new_token                   │
└─────────────────────────────────────────┘
```

### 6.3 数据拉取策略

**不拉原始日志，只拉元数据**：

| 数据类型 | 获取方式 | 用途 |
|---------|---------|------|
| 信号元数据 | API查询（时间范围+信号名） | 图表展示 |
| 告警阈值 | 配置中心（启动时加载） | 异常判断 |
| 历史统计 | API聚合接口（若提供） | 周报使用 |

### 6.4 异常检测逻辑

```
信号数据拉取后：
        │
        ▼
  遍历配置的信号阈值（来自D5决策）
        │
        ├── 纵向加速度 | 超出 ±X g → 疑似MRC
        ├── 方向盘转角 | 超出 ±Y° → 疑似失效
        ├── VCAN总线状态 | 错误计数>Z → 数据质量告警
        └── 接管请求信号 | 存在 → 记录接管场景
                │
                ▼
        推送给数据负责人（模块4）
                │
                ▼
        人工确认 → 建缺陷 / 标记噪声+调阈值
```

### 6.5 静态图表生成

```
Python matplotlib 流程：
  1. DR API → 获取时间序列数据
  2. matplotlib.figure.Figure
  3. plot() → 折线/柱状
  4. savefig(format='png', dpi=150)
  5. 图片路径 → 飞书消息 attachments

图表类型：
  - 折线图：信号随时间变化
  - 柱状图：某时段内异常次数统计
  - 热力图：多信号相关性（选做）
```

## 7. P0 架构决策（D1-D5）

### D1: Bitable 并发写入策略

**问题**：多个用户同时操作用例，机器人写入是否会冲突？

**推荐方案**：**机器人只写特定字段，读写全字段**

```
机器人可写字段（白名单）：
┌────────────────┬────────────────────────────────┐
│ 字段            │ 说明                            │
├────────────────┼────────────────────────────────┤
│ 执行状态        │ 待执行→执行中→通过/失败/阻塞     │
│ 执行日期        │ 最近一次执行时间                  │
│ 执行人          │ 谁执行                          │
│ 关联场景ID      │ 挂住的场景ID列表                  │
│ 最近更新人      │ 谁最后修改                       │
│ 备注            │ 可追加简短说明                   │
└────────────────┴────────────────────────────────┘

机器人只读字段（不回写）：
  用例ID、用例名称、测试类型、功能模块、关联需求、优先级、用例版本

用户直接编辑字段：
  执行状态、执行人、执行日期（用户自己改，机器人也可改）
```

**冲突处理**：
- Bitable单字段原子更新，不锁表
- 并发冲突：后者覆盖前者（飞书Bitable API特性）
- 高频写入场景（如批量更新执行状态）：消息队列串行化

### D2: 对话上下文持久化边界

**问题**：30分钟快照是否足够？什么情况下会丢上下文？

**推荐方案**：**30分钟基准 + 三重保护**

| 触发条件 | 行为 | 丢上下文场景 |
|---------|------|------------|
| 10轮对话 | 立即快照 | ❌ 不丢 |
| 30分钟无交互 | 快照+休眠 | ❌ 不丢 |
| 进程重启 | 从Bitable恢复 | ❌ 不丢 |
| 用户关闭飞书 | 快照+结束 | ❌ 不丢 |
| Bitable API故障 | 内存降级，告警 | ⚠️ 可能丢（仅M1期间可接受）|
| 服务器断电 | 最多丢30分钟 | ⚠️ 可接受 |

**结论**：30分钟是合理的平衡点。测试工程师的平均操作间隔通常>15分钟，30分钟可以覆盖绝大多数工作间隙场景。

### D5: DR平台 API 获取时间和 Buffer 计划

**问题**：DR数据什么时候拉取？实时还是定时？

**推荐方案**：**混合策略（定时+实时+Buffer）**

```
1. 定时批量拉取（离线分析用）
   - 每周五拉取本周数据（周报使用）
   - 每周一拉取上周数据（确保完整性）
   - 时间窗口：T-7天 到 T-1天（T=今天）

2. 实时查询（用户主动请求）
   - 用户问"ICC最近的CAN总线有没有异常"
   - 默认查最近24小时
   - 可配置时间范围
   - 结果缓存5分钟（Redis key: dr:query:{hash}）

3. DR平台数据Buffer
   - DR数据产出 vs 用户可查询之间有延迟
   - 推荐：查询时间窗口向后扩10分钟
   - 示例：用户查"过去1小时"，实际拉 "1小时+10分钟"
   - 原因：DR平台数据写入可能有分钟级延迟

4. 直播场景（选做）
   - 用户说"帮我盯着DR数据"
   - 每5分钟轮询一次，结果暂存
   - 用户说"停止"时停止轮询
```

**DR API调用频率估算**：

| 场景 | 频率 | 备注 |
|-----|------|------|
| 周报生成 | 1次/周 | 手动触发 |
| 日常查询 | ~50次/天 | 工程师主动查询 |
| 图表生成 | ~20次/天 | 每次1-3张图 |
| **合计** | **~70次/天** | **低于API限额** |
